## Background

The `Product naming` and `Credential file naming` themes in [docs/version-v0.3.md](docs/version-v0.3.md) require a repo-wide naming transition while preserving operator clarity. You already updated Make targets on this branch; this plan treats those changes as baseline and focuses on remaining symbol classes, path/package validation, and staged implementation.

**Related docs:** [docs/version-v0.3.md](docs/version-v0.3.md), [docs/packaging.md](docs/packaging.md), [docs/configuration.md](docs/configuration.md), [README.md](README.md).

---

## Phase 1 — Discovery and symbol inventory

**Goal.** Produce a high-signal inventory of where naming, binaries, config/data paths, and package manifests are defined, then lock a migration matrix before edits.

**Deliverables**

- Build a **symbol taxonomy inventory** (broad/representative, not exhaustive) across first-party paths: `cmd/`, `internal/`, `docs/`, `scripts/`, `config/`, `.github/`, release manifests.
- Capture naming classes and current forms:
  - Product terms in docs/UI/log copy (`Chimera`, `Chimera`, `Porcelain`, `Locus`).
  - Env/flags/headers/constants (`CHIMERA_*`, `X-Chimera-*`, command help text).
  - Binary/package identifiers (`chimera`, `chimera-indexer`, desktop binary names, archive names).
- Build a **binary/artifact location map** from source-of-truth files: [Makefile](Makefile), [scripts/chimera-names.sh](scripts/chimera-names.sh), [scripts/desktop-build.sh](scripts/desktop-build.sh), [scripts/release-package.sh](scripts/release-package.sh), [.goreleaser.yaml](.goreleaser.yaml).
- Build a **runtime path map** for config/state/data/log/run directories from [internal/config/config.go](internal/config/config.go), [internal/indexer/config.go](internal/indexer/config.go), [config/gateway.example.yaml](config/gateway.example.yaml), [config/indexer.example.yaml](config/indexer.example.yaml), [scripts/clean-data.sh](scripts/clean-data.sh), [.gitignore](.gitignore).
- Perform package validation passes:
  - Release archive contract (`.goreleaser.yaml` contents and names).
  - Personal bundle contract (`scripts/release-package.sh` outputs under `dist/personal/`).
  - Check expected files/paths in packaged outputs against [docs/packaging.md](docs/packaging.md).
- Produce a **rename decision matrix** (what renames now vs deferred) keyed by symbol class.

**Acceptance**

- One migration matrix exists with concrete symbol classes, owners, and target names.
- Binary output, archive naming, and runtime path behavior are documented with source-file references.
- Unknowns are reduced to explicit, bounded decisions (no hidden assumptions before implementation).

**Status:** `done`

---

## Phase 2 — Constants and docs/comment rename

**Goal.** Centralize product naming constants and apply terminology updates to docs/comments in controlled batches.

**Deliverables**

- Define/normalize global naming constants where appropriate (shell/build/release/help-text shared constants first).
- Ensure binary/display naming sources are minimally duplicated (single-source where practical).
- Rename docs and comments to layered terminology in operator-first order:
  - Core docs: [README.md](README.md), [docs/configuration.md](docs/configuration.md), [docs/packaging.md](docs/packaging.md), version/plan references.
  - In-code comments/help text: [cmd/chimera/main.go](cmd/chimera/main.go), [cmd/chimera/gateway.go](cmd/chimera/gateway.go), [cmd/chimera/serve.go](cmd/chimera/serve.go), [cmd/chimera-index/main.go](cmd/chimera-index/main.go).
- Use scripted find/replace where safe (PowerShell), then manually review semantic edge cases.

**Acceptance**

- Naming constants are discoverable and used from a minimal set of source locations.
- Top-level docs and command help no longer present legacy product naming as primary identity.
- Comment/doc updates avoid changing behavior and preserve operator instructions.

**Status:** `done`

---

## Phase 3 — Migration and operational surfaces

**Goal.** Apply hard-cut naming to migrations/config/data/rules/CI/assets and align packaging/runtime contracts.

**Deliverables**

- Apply hard-cut rename plan for operational interfaces selected in Phase 1 matrix:
  - Config keys/files/path docs and generated examples.
  - Data/state directory names and cleanup scripts.
  - Cursor rules/docs references and GitHub workflow/release metadata.
  - Static assets/UI text where naming is operator-visible.
- Update packaging/release definitions to target names and verify generated archive/bundle layouts.
- Add migration notes for operators where paths/file names change (explicit “from → to”).

**Acceptance**

- Config/data/runtime directories and package artifacts match updated docs and scripts.
- CI/release metadata no longer emits legacy naming where in scope.
- Operator migration instructions exist for every breaking path/name change.

**Status:** `done`

---

## Phase 4 — Go source and tests

**Goal.** Complete runtime code and test-suite rename updates so behavior, APIs, and test fixtures are coherent.

**Deliverables**

- Rename remaining Go symbols/string literals/log text in gateway/indexer code paths with attention to:
  - HTTP header constants and request handling.
  - Environment variable lookups and startup config resolution.
  - Startup logs and UI/API-facing response text.
- Update tests/fixtures/snapshots to new naming and path expectations (unit + integration surfaces).
- Run targeted test groups around renamed surfaces (config resolution, API handlers, indexer client, packaging checks if scripted).

**Acceptance**

- Go build/test passes for affected areas with updated naming.
- No stale legacy naming remains in high-traffic runtime/test surfaces identified in Phase 1.
- Final matrix is checked off with implemented vs intentionally deferred entries.

**Status:** `done`

## Phase 5 — Markdown documentation rename pass

**Goal.** Complete the coordinated markdown-only rename pass after code/config/packaging surfaces stabilize.

**Deliverables**

- Update Markdown naming across `README`, `docs/`, and plan docs to current implemented contracts.
- Refresh command examples, binary names, paths, env vars, and header names for consistency.
- Keep legacy names only where they are explicitly migration aliases (not primary identity).

**Acceptance**

- Operator-facing docs no longer present legacy names as primary.
- Docs examples match current build/package outputs and supported aliases.

**Status:** `todo`

## Phase 6 — Documentation consolidation

**Goal.** Normalize documentation after the rename pass and clearly separate historical context from current behavior.

**Deliverables**

- Resolve stale historical statements in planning docs that still read as present-state mismatches.
- Add explicit historical labels where old behavior is retained for reference.
- Ensure docs reference the same canonical build/run/package contracts and entrypoint paths.

**Acceptance**

- No contradictory docs for current runtime/packaging behavior.
- Historical notes are explicitly marked and not mistaken for current requirements.

**Status:** `done`

## Phase 7 — Legacy alias policy lock

**Goal.** Define final compatibility policy for legacy names and enforce it through docs/tests.

**Deliverables**

- Decide keep/remove timelines for legacy env/header/file aliases.
- Document deprecation windows and operator migration expectations.
- Add/adjust targeted tests for retained alias behavior and removal boundaries.

**Acceptance**

- Compatibility matrix is explicit and internally consistent.
- Alias behavior in code, tests, and docs matches policy.

**Status:** `done` (hard-cut policy: no legacy alias support)

- Remaining legacy names (if any) are explicitly intentional.
